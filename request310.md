产品需求与技术架构文档 (PRD)项目名称： 智能四象限时间管理 App (暂定名)目标平台： Android & iOS 跨平台核心理念： 艾森豪威尔矩阵 (宏观优先级) + 24小时时间轴 (微观执行) + 大模型语音解析 (极简输入)1. 技术栈选型 (Tech Stack)前端框架： Flutter (Dart 语言)开发环境： VS Code / Android Studio状态管理： Riverpod (或 Bloc) —— 要求严格保证三大模块的数据同步响应。本地数据库： Isar 或 Hive —— 要求支持复杂的查询与高频的拖拽状态持久化。AI/NLP 模块： 系统原生 speech_to_text + 外部大语言模型 API (如 Gemini/通义千问)。2. 核心数据模型 (Data Model)所有的业务逻辑都围绕 PlanEvent 实体展开。字段名类型说明必填idString唯一标识符 (UUID)是titleString任务名称 / 计划内容是startTimeDateTime?开始时间。为空时表示任务仅停留在四象限池中，尚未排期否endTimeDateTime?结束时间 (默认时长可设为1小时)否locationString?地点信息 (由语音提取)否isImportantbool是否重要 (决定归属的象限)是isUrgentbool是否紧急 (决定归属的象限)是isCompletedbool是否已打卡/完成是 (默认 false)3. 核心功能模块划分 (BottomNavigationBar)采用底部三 Tab 导航架构，确保用户在宏观规划和微观执行之间无缝切换。模块一：四象限看板 (Quadrant Screen)功能定位： 任务的“收集箱”与“优先级定义区”。UI 呈现： 2x2 网格视图 (GridView)。分为：重要且紧急（红）、重要不紧急（蓝）、紧急不重要（黄）、不重要不紧急（灰）。数据逻辑： 只展示 startTime 为 null 的任务。交互设计： 点击卡片可手动编辑详情或分配日期。模块二：日历视图 (Calendar Screen)功能定位： 宏观时间概览。UI 呈现： 月/周视图（基于 table_calendar）。数据逻辑： 遍历数据库中带有 startTime 的任务。在对应日期下方绘制代表象限颜色的标记点（Marker）。交互设计： 点击具体某一天，路由跳转或传参切换至“模块三”，查看该日详情。模块三：24小时执行表与排期 (Daily Timeline Screen)功能定位： 当日任务落地与微观时间调度。UI 呈现：主视图： 当日 00:00 - 24:00 的纵向时间轴。每个小时区块作为一个接收区 (DragTarget)。待办抽屉 (Task Drawer)： 侧滑栏或底部弹窗，展示尚未排期的四象限任务。核心交互 (拖拽排期)：抽屉内的任务包装为 Draggable。用户将任务从抽屉拖拽到主视图的特定时间块并松手。触发 onAccept 回调：更新该任务的 startTime，写入数据库，页面状态刷新，任务从抽屉消失并固定在时间轴上。4. 特色功能：智能语音调度 (Smart Voice Input)业务流程：语音采集： 用户点击全局悬浮按钮 (FAB)，长按录音。STT 转换： 调用 speech_to_text 将语音转为文本（如：“明天下午3点在图书馆复习高数”）。大模型解析： 将文本拼接 Prompt 发送至 LLM API。结构化输出： LLM 返回标准 JSON（自动解析出 startTime: "2026-03-11T15:00:00", location: "图书馆", 并根据语义自动判断 isImportant: true, isUrgent: false）。拦截确认： 弹出底窗让用户核对 AI 解析结果，用户点击“确认”后存入本地数据库。5. 开发里程碑 (Milestones)为了保证项目的顺利推进，建议分阶段进行开发：Phase 1: 静态骨架 (UI & 路由)搭建跨平台工程，配置依赖。实现 BottomNavigationBar 以及三个页面的基础静态 UI（先用假数据渲染）。Phase 2: 拖拽与底层交互 (Drag & Drop)在 24 小时模块中引入 Draggable 和 DragTarget。跑通“从抽屉拖拽到时间轴改变时间”的纯内存逻辑。Phase 3: 状态管理与数据持久化 (State & DB)引入 Riverpod 和 Isar 数据库。实现真实数据的增删改查，确保一个模块修改数据，其他模块瞬间响应。Phase 4: 智能化接入 (AI Integration)集成语音识别库。调试大模型 Prompt，完成 JSON 数据的解析与反序列化入库。
