智能四象限时间管理 App - 前端与架构需求文档 (PRD)
1. 整体架构与部署逻辑 (部署要求)
本项目采用**“离线优先的双擎架构 (Offline-First Dual-Engine)”**，在保证用户隐私和无网可用性的前提下，利用大模型提升复杂场景的解析上限。这套数据处理流水线将作为核心中枢运行在客户端。

1.1 输入层：统一智能舱 (Smart Input)
输入源整合： 无论是文本输入（键盘）还是语音输入（通过系统原生 speech_to_text），最终都转化为标准的 String 字符串进入处理管道。

1.2 处理层：双擎解析流水线 (Dual-Engine Pipeline)
引擎一（主引擎）：本地启发式规则引擎 (Local Heuristic Engine)

部署位置： 纯客户端本地计算，零网络延迟。

逻辑： 运行预设的正则表达式（正则捕获时间特征：明天、14点）和关键词匹配（匹配象限特征：紧急、开会）。

出口： 如果正则匹配置信度高（成功提取时间和有效标题），则直接生成 PlanEvent 结构体，跳过网络请求。

引擎二（备用引擎）：大模型云端解析 (LLM Fallback)

触发条件： 当本地引擎无法解析时间（例如输入“下个礼拜五下班后”），或提取出的关键信息缺失时触发。

部署要求： 接入免费的轻量级 API（如 Gemini Flash 等）。

降级策略： 如果发生网络超时或 API 报错，自动降级为“手动创建表单”，将用户输入的原始字符串填入标题框。

1.3 拦截与存储层
用户确认 (Human-in-the-loop)： 解析完成后，严禁直接静默入库。必须弹出 BottomSheet 供用户核对时间、象限和标题，允许手动微调。

本地存储： 确认后，数据写入本地数据库（如 Isar 或 Hive）。通过 Riverpod 状态管理，触发全应用 UI 层的响应式刷新。

2. 前端界面与交互要求 (UI/UX)
应用采用底部导航栏 (BottomNavigationBar) 架构，分为三个核心业务模块，外加一个全局悬浮的智能输入舱。

2.1 全局组件：智能输入舱 (Smart Input Bar)
位置： 悬浮于所有页面的底部，或作为主要的全局 FAB（悬浮操作按钮）展开后的形态。

UI 构成： 左侧为单行/多行 TextField，右侧为麦克风图标。

交互动作：

点击输入框： 唤起键盘，用户可直接打字，键盘回车键绑定发送动作。

长按麦克风： 触发录音动画及系统震动反馈，松手即停止录音并转化为文字填入左侧输入框，随后自动触发双擎解析逻辑。

2.2 模块一：四象限看板 (优先级梳理区)
视觉呈现： 2x2 的全屏网格视图 (GridView)。

第一象限： 重要且紧急（红色调 / 警示）

第二象限： 重要不紧急（蓝色调 / 规划）

第三象限： 紧急不重要（黄色调 / 委托）

第四象限： 不重要不紧急（灰色调 / 搁置）

数据过滤： 仅展示数据库中 startTime 为 null（即未分配具体执行时间）的待办任务。

交互逻辑： 点击任务卡片可查看详情或手动指定日期，指定日期后该卡片从当前视图消失，流入日历或时间轴。

2.3 模块二：日历视图 (宏观规划区)
视觉呈现： 经典的月历/周历切换视图。

数据渲染： 遍历所有已分配 startTime 的任务。在日历的具体日期单元格下方，使用对应象限颜色的微小圆点作为标记（Marker）。

交互逻辑： 点击特定日期，应用自动跳转（或传递参数）至“模块三”，并展示选定日期的 24 小时时间轴。

2.4 模块三：24小时时间轴 (微观执行区)
此模块是执行排期的核心战场，要求实现极度流畅的拖拽体验。

区域 A：当日时间轴 (主视图)

呈现： 00:00 至 24:00 的纵向滚动列表。

底层逻辑： 每个小时（或半小时）区块必须包裹为 DragTarget<PlanEvent>，作为可接收任务的降落区。悬停时需有视觉高亮反馈。

区域 B：待办抽屉 (Task Drawer)

呈现： 侧滑菜单或底部可拉出的抽屉。

数据： 展示来自“模块一”的、未分配时间的四象限任务。

底层逻辑： 抽屉内的每一个任务卡片必须包裹为 Draggable<PlanEvent>。

核心交互 (拖拽排期)： 用户从抽屉中按住卡片拖动，卡片变为半透明悬浮状态 (feedback)。拖至主视图的特定时间块松手触发 onAccept，更新任务的 startTime 并落库，抽屉内卡片消失，时间轴对应位置生成任务区块。
